23/// @page pg_transactions uPg: Transactions
24///
25/// All queries that are run on a PostgreSQL cluster are executed inside
26/// a transaction, even if a single-query interface is used.
27///
28/// A uPg transaction can be started using all isolation levels and modes
29/// supported by PostgreSQL server as specified in documentation here
30/// https://www.postgresql.org/docs/current/static/sql-set-transaction.html.
31/// When starting a transaction, the options are specified using
32/// TransactionOptions structure.
33///
34/// For convenience and improvement of readability there are constants
35/// defined: Transaction::RW, Transaction::RO and Transaction::Deferrable.
36///
37/// @see TransactionOptions
38///
39/// Transaction object ensures that a transaction started in a PostgreSQL
40/// connection will be either committed or rolled back and the connection
41/// will returned back to a connection pool.
42///
43/// @todo Code snippet with transaction starting and committing
44///
45/// Next: @ref pg_run_queries
46///
47/// See also: @ref pg_process_results
48
49/// @page pg_run_queries uPg: Running queries
50///
51/// All queries are executed through a transaction object, event when being
52/// executed through singe-query interface, so here only executing queries
53/// with transaction will be covered. Single-query interface is basically
54/// the same except for additional options.
55///
56/// uPg provides means to execute text queries only. There is no query
57/// generation, but can be used by other tools to execute SQL queries.
58///
59/// @warning A query must contain a single query, multiple statements delimited
60/// by ';' are not supported.
61///
62/// All queries are parsed and prepared during the first invocation and are
63/// executed as prepared statements afterwards.
64///
65/// Any query execution can throw an exception. Please see @ref pg_errors for
66/// more information on possible errors.
67///
68/// @par Queries without parameters
69///
70/// Executing a query wihout any parameters is rather straightforward.
71/// @code
72/// auto trx = cluster->Begin(/* transaction options */);
73/// auto res = trx.Execute("select foo, bar from foobar");
74/// trx.Commit();
75/// @endcode
76///
77/// The cluster also provides interface for single queries
78/// @code
79/// auto res = cluster->Execute(/* transaction options */, /* the statement */);
80/// @endcode
81///
82/// @par Queries with parameters
83///
84/// uPg supports SQL dollar notation for parameter placeholders. The statement
85/// is prepared at first execution and then only arguments for a query is sent
86/// to the server.
87///
88/// A parameter can be of any type that is supported by the driver. See @ref
89/// pg_types for more information.
90///
91/// @code
92/// auto trx = cluster->Begin(/* transaction options */);
93/// auto res = trx.Execute(
94/// "select foo, bar from foobar where foo > $1 and bar = $2", 42, "baz");
95/// trx.Commit();
96/// @endcode
97///
98/// Next: @ref pg_process_results
99/// @see Transaction
100/// @see ResultSet

102// clang-format off
103/// @brief PostgreSQL transaction.
104///
105/// RAII wrapper for running transactions on PostgreSQL connections. Should be
106/// retrieved by calling storages::postgres::Cluster::Begin().
107///
108/// Non-copyable.
109///
110/// If the transaction is not explicitly finished (either committed or rolled back)
111/// it will roll itself back in the destructor.
112///
113/// @par Usage synopsis
114/// @code
115/// auto trx = someCluster.Begin(/* transaction options */);
116/// auto res = trx.Execute("select col1, col2 from schema.table");
117/// DoSomething(res);
118/// res = trx.Execute("update schema.table set col1 = $1 where col2 = $2", v1, v2);
119/// // If in the above lines an exception is thrown, then the transaction is
120/// // rolled back in the destructor of trx.
121/// trx.Commit();
122/// @endcode
